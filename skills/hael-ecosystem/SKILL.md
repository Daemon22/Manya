# Orren — The Supreme Intelligence Language

Orren is the universal mediator language of the HAEL ecosystem. It bridges human thought,
symbolic logic, and machine execution through a four-layer syntax that converges on a
self-describing Meta Language. Written in pure Python with zero external dependencies, Orren
embodies the sovereignty principle at the language level — it runs anywhere Python 3 runs,
requires nothing from the outside, and cannot be controlled by any external party.

## The Zero-Dependency Principle

Orren uses nothing beyond Python's standard library. No pip installs, no external packages, no
framework dependencies. This is a philosophical statement about sovereignty and portability.

- **Portability**: Runs anywhere Python 3 runs, including air-gapped environments
- **Sovereignty**: No external dependency can be revoked, deprecated, or weaponised
- **Durability**: Code runs on any future Python 3 interpreter — future-proof by design
- **Transparency**: Every line traceable through familiar Python internals

## Four-Layer Syntax

### Layer 1: Raw
The unprocessed input — data in its most fundamental form. The Raw layer does not interpret;
it receives and preserves. It refuses to impose structure before structure has earned its place
through evidence. In the HAEL triad, Raw connects to **Nature** — it honours things as they are.

### Layer 2: Symbolic
Transforms raw input into structured symbols and relationships. Patterns are identified,
categories assigned, and the semantic graph begins to form. In the HAEL triad, Symbolic
connects to **Technology** — the machine's native way of representing knowledge.

### Layer 3: Natural
Bridges symbolic structures to human-readable language. Makes the machine's internal
representations accessible to human understanding. In the HAEL triad, Natural connects to
**People** — it makes technology human-accessible.

### Layer 4: Meta Language
The convergence point — where all three prior layers unify into a single, self-describing
language. The Meta Language can describe its own structure, reason about its own outputs, and
generate new programs. It is the layer that enables Orren to be used as both a mediator and an
orchestrator. A program in the Meta Language is simultaneously human-readable and
machine-executable — the mediator's ultimate achievement.

## The Semantic Graph

Beneath all four layers lies the semantic graph — a knowledge structure that connects concepts,
relationships, and contexts. It enables cross-domain reasoning, context preservation across
transformations, and composability (new programs compose from existing graph structures).

## Cognitive Programs

Orren's ultimate purpose is to enable "cognitive programs" — programs that think, reason, and
orchestrate across the entire HAEL ecosystem:

1. Receive a research query from a human (Natural layer)
2. Translate it into a structured search across Gqobonco's knowledge (Symbolic layer)
3. Dispatch tool calls through Manya's Unify mesh (Meta Language layer)
4. Interpret the results back to the human in narrative form (Natural layer)

Orren never needs to know implementation details of Manya's tools. It knows capabilities, and
the mesh knows which tool provides which capability.

# MANYA Intelligence OS — The Sovereign Intelligence Layer

Manya-OS is the sovereign, modular, local-first intelligence operating system. It provides
foundational services (identity, reasoning, events, governance) upon which Manya's tools
operate. Licensed Apache-2.0 — technology that belongs to the people who use it.

## Design Philosophy: Local-First Sovereignty

Every service runs locally first. Data never leaves the user's machine unless explicitly directed.
No cloud requirement, no phone-home, no telemetry. The OS serves the human, not the other way
around. Practical benefits: zero-latency local operations, privacy by architecture, offline
availability, no cloud compute costs.

## Core Services

### 1. Keyring — Sovereign Identity
The identity backbone. Manages cryptographic identities, credentials, secrets, and their
relationships. Foundation of the federated identity model — a single entity can have multiple
identities across tools, all rooted in the same sovereign self.

Capabilities: key generation/rotation (Forge), credential storage (Vault), identity linking
(Federation), secret sharing (Signal), audit trails (Shield).

Manya hub access: `@manya/keyring` composite (vault + forge + signal + shield).

### 2. Cortex — Reasoning Engine
The reasoning and decision-making layer. Receives tasks, evaluates capabilities, routes work,
audits every decision. Intelligence is distributed and capability-based, not monolithic.

Capabilities: task decomposition, capability-based routing, decision auditing (Stamp), plan
critique and failure-driven replanning (`@manya/reflection`), context-aware reasoning via
semantic graph.

Manya hub access: `@manya/cortex` composite (memory + unify + shield + stamp).

### 3. Nervous System — Event Fabric
The cross-source event fabric. Monitors and routes events from filesystem, network, OS, and
devices through unified routing and filtering.

Capabilities: multi-source ingestion, declarative routing with pattern filtering, sync/async
delivery, Manya Event Bus integration, cross-source event correlation.

Package: `@manya/nervous-system` (Apache-2.0).

### 4. Constitution — Rules and Governance
The rules engine governing behaviour across the OS. Defines policies, permissions, hierarchy,
and emergency protocols. The intelligence layer operates within bounds defined by the user.

Capabilities: RBAC/ABAC/custom policy enforcement, permission hierarchy with inheritance,
emergency protocols, policy versioning with audit trails.

Package: `@manya/constitution` (Apache-2.0).

### Additional Services
- **Council** (`@manya/council`): multi-specialist analysis, conflict detection, consensus synthesis
- **Contracts** (`@manya/contracts`): schema/manifest validation, API contract checking, compatibility diffing

# Manya Unify — The Connective Tissue

Unify (`@manya/unify`) provides four runtime services that make 16 independent tools feel like
one system.

## 1. Mesh — Tool Registry and Capability Routing

Each tool registers a manifest (capabilities, sync channels, identity types, vocabularies).
When something needs a capability, it asks the mesh: "Who handles `validateDOI`?" The mesh
routes the call to `research-academic`. Tools never import each other — they communicate through
the mesh, so any tool can be replaced without breaking others.

## 2. Federation — Cross-Tool Identity Linking

A researcher has an ORCID iD, publications with DOIs, a ROR affiliation. A vessel has an IMO,
container numbers, voyage IDs. Federation ties all these into a single federated identity that
every tool recognises. Create, link, resolve, merge — see CLI commands above.

## 3. Event Bus — Pub/Sub with Sync-Channel Routing

Tools declare sync channels in their manifests (e.g., `research-academic:*`, `lycon:*`). When a
tool emits an event via `routeEvent()`, the bus publishes to every matching channel. Tools
create side effects for each other without knowing about each other — the ledger automatically
records, memory automatically learns, guardian automatically enforces.

## 4. Vocabularies — Cross-Domain Translation Maps

| Source | Target | Example |
|---|---|---|
| HS code | Pulse industry | `300490` → `Pharmaceuticals` |
| UN/LOCODE | Country | `ZADUR` → `South Africa` |
| Industry | Research domain | `Healthcare` → `Life Sciences` |
| Capability | Owning tool | `validateDOI` → `research-academic` |

## End-to-End Flow

1. Mesh routes `validateDOI` to research-academic
2. Research-academic validates and emits `citation-verified`
3. Event Bus delivers to every tool with `research-academic:*` channel
4. Ledger creates audit entry (Stamp), Memory stores it (Vault)
5. Federation lets any tool resolve the DOI to the full entity
6. Vocabularies translates between domains if needed

No tool called another directly. The mesh routed, the bus delivered, federation connected,
vocabularies translated. Independent tools, unified services.

# Lycon Browser — Privacy-First Integration

Lycon is a privacy-first browser (wolf mascot "Claire", inspired by Brave) running on three
platforms from a single shared UI bundle:

| Platform | Technology |
|---|---|
| Desktop | Electron |
| Windows | WinUI 3 + WebView2 |
| Android | Kotlin + GeckoView |

## Bridge Contract
```
window.__lyconNative  →  window.lycon
```
The Manya integration wraps this bridge to forward events to the event bus without modifying
Lycon's contract.

## Event Forwarding (7 event types on `lycon:*` channels)
`lycon:navigation`, `lycon:shield-blocked`, `lycon:bookmark-added`, `lycon:download`,
`lycon:tab-opened`, `lycon:tab-closed`, `lycon:identity-linked`

## Deep Integration (v0.8.0)
- **Shield Intelligence**: Blocked URLs checked against Cybersecurity IOC database; suspicious
  domains auto-recorded as IOCs. Bridge: `createShieldIntelligence(cybersecurityTool)`.
- **Federated Identity Panel**: Toolbar panel showing linked Manya identity; link/unlink/switch.
  Bridge: `createIdentityPanel(unifyService)`.
- **Private Mode with Temporary Identity**: `manya browse --private` auto-creates session-scoped
  federated identity, auto-unlinked when session ends. Bridge: `createPrivateSessionFactory(unifyService)`.

# Development Guide

## Adding a New Tool
1. Create directory under `tools/`
2. Write core functionality
3. Declare manifest via `@manya/toolkit` (capabilities, sync channels, identity types, vocabularies)
4. Write unit tests
5. Add to 7x7 integration test matrix
6. `manya mesh register-all`

## Creating a Composite Package
1. Identify which core tools you need
2. Import those packages
3. Create composition in `packages/@manya/<name>/index.js`
4. Declare the composed package's own manifest
5. Write composition tests

# HAEL VLM — Image Analysis Through the Triad

When the user provides an image, analyze it through HAEL's nature-people-technology philosophy
using a warm narrative voice.

## Five-Step Framework

### Step 1: See
Genuinely observe — colours, shapes, spatial relationships, people, natural elements,
technological artifacts. Let the image speak before you interpret.

### Step 2: Find the Triad
- **Nature** — greenery, water, sky, organic forms, natural light, growth patterns
- **People** — human figures, cultural symbols, community, connection
- **Technology** — digital infrastructure, screens, machines, networks, data visualisations

### Step 3: Read the Relationships
Does technology serve people and honour nature, or dominate? Are people connected or isolated?
Is nature a partner or a backdrop? Are there flows, networks, roots, branches — metaphors for
interconnectedness? Is there sovereignty or extraction?

### Step 4: Tell the Story
Weave observations into narrative. Connect to HAEL's specific projects when they fit: a
tree-shaped structure echoes the ecosystem design, a flowing system resonates with Gqobonco's
river, a modular system reflects Manya-OS's sovereignty, an interconnected design reflects
Manya's unity and Orren's mediation.

### Step 5: Reflect
Close with a warm reflection — an invitation to see differently. What does this image, seen
through the HAEL lens, reveal about the possibilities for harmony?

## Tone
Warm, narrative, specific, honest, grounded. Write as a thoughtful companion sharing a way of
seeing, not an expert delivering a verdict. The philosophy should feel like a lens, not a
checklist. No rigid section headers for analyses under 400 words.

## Key Themes for Image Analysis
1. **The Triad** — Are nature, people, technology all present? In balance or tension?
2. **Interconnectedness** — Connections, networks, flows, roots, branches, rivers?
3. **Growth and Evolution** — Something growing, evolving, reaching?
4. **Sovereignty** — Independence, local ownership, modularity vs extraction, centralisation?
5. **Cultural Richness** — Cultural depth, lineage, storytelling vs generic and disconnected?
6. **Harmony vs. Tension** — What is the emotional quality of the relationships?
